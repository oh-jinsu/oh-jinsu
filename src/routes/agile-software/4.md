MVVM 아키텍처의 단점을 해결하기 위해 고민이 많았다. 프로덕션 목적의 개인 프로젝트를 진행하는 동시에 새로운 설계를 시도했다. 설계는 실제로 맞닥뜨릴 법한 구체적인 요구 사항을 수용할 수 있어야 하기 때문이다. 머리를 굴리기보다는 손을 움직여야 한다.

지금까지 진척된 설계를 그대로 애플리케이션을 마감하더라도 보이는 데에는 큰 무리가 없을 것이다. 하지만 보이지 않은 코드의 품질에 관해서는 개선이 필요하다. 실험적이거나 임기응변적인 코드가 적지 않으며 개념도 명료한 수준이 아니다. 마무리하기 전에 우선 할 일이 남아 있다.

시도한 설계는 현대 애플리케이션의 일반적인 요구에 부응하는 동시에 기왕의 아키텍처의 결함을 보완하는 데에 의의를 두었다. 이 글에서는 그러한 일반적인 요구를 기술적인 언어로 번역한다. 개선에 앞서서 한층 뚜렷한 개념을 획득할 수 있을 것이다.

Redux에 대해 짚고 넘어가야겠다. 유명세에도 불구하고 그전까지 자세히 들여다 보지 않았다. 심지어 설치해 본 적도 없다. 그동안 고유한 필요성을 전혀 느끼지 못한 까닭이다. 그런데 이번에 내 나름으로 시도한 설계가 Redux와 매우 흡사한 모습이라는 사실을 뒤늦게 알아차리게 되었다. 개선에 앞서서 지금껏의 설계를 캡쳐해 놓았으니 [여기에서](https://github.com/oh-jinsu/album-captured) 확인할 수 있다.

"React의 Context API와 다를 게 뭐람." 그런 때도 있었다. 이제야 비로소 Redux의 핵심 개념을 이해했다. Store, Reducer, 혹은 Action, 특히 번잡스럽기만하던 Middleware의 설계 의도에 대해서 말이다. Redux는 내가 혼자서 고민하던 많은 문제들에 대해 이미 세련된 해결책들을 내놓고 있었다.

따라서 나는 글에서 Redux의 개념들을 적극적으로 활용하고 필요하다면 기꺼이 받아들일 것이다.

## 트리 구조
애플리케이션의 구조는 단순하지 않다. 우선 애플리케이션을 최상단에 자리하는 노드처럼 생각해 보자. 웹 애플리케이션이나 모바일 애플리케이션이나 마찬가지로 그 아래에는 독립적인 화면들이 있다. 그리고 화면 아래에도 독립적인 컴포넌트들이 있다. 네비게이션 컴포넌트와 리스트뷰 컴포넌트는 분명 동질적이지 않다.
```
+-- application
|   +-- home
|   |   +-- navigation
|   |   +-- list_view
|   |   .
|   |   .
|   |   .
|   +-- profile
|   .
|   .
|   .
```
중요한 건 각각의 노드마다 상태를 가지고 있다는 사실이다. 이를테면 가장 말단의 `list_view` 컴포넌트는 아이템의 리스트에 대한 상태를 지니고 있어야 한다. `navigation` 컴포넌트도 현재 위치에 대한 정보를 지니고 있다면 좋을 것이다. 그런데 위치 정보라면 `navigation` 컴포넌트 말고도 다른 컴포넌트도 사용해야 할 수 있다. 그렇다면 상태를 지엽적인 `navigation`에 두기 보다는 `application`에 위치시키는 편이 나을 것이다.

이때 상태의 저장소를 Redux를 따라 Store라고 부르자.

> 설계 원칙 1. 하나의 노드는 여러 개의 스토어를 가질 수 있다.

## 이벤트의 전파
소프트웨어의 트리 구조는 모바일 애플리케이션에서 더욱 체감된다. 화면의 스택이 쌓이기 때문이다. 맨 밑에는 애플리케이션이, 그 위에는 이를테면 홈 화면이 있고, 그 위에는 다른 화면이 또 쌓일 수 있다. 현재 진행하고 있는 앨범 프로젝트가 그렇다. 홈 화면에는 앨범의 리스트가 있다. 아이템을 탭하면 하나의 앨범에 대한 화면이 위로 쌓인다.
```
+-- application
|   +-- home
|   |   +-- navigation
|   |   +-- album_list
|   +-- album
|   .   +-- dialog_to_add_photo
|   .   .
|   .   .
```
`album`의 `dialog_to_add_photo`는 사진을 추가하기 위해 마련한 독립적인 자식 컴포넌트다. 사진을 추가했을 때 `album`은 업데이트되어야 한다. 그런데 이때 `album_list`에 속한 특정한 앨범의 커버 이미지도 업데이트되어야 한다면 어떨까? 이때 이벤트는 어떻게 전파될 수 있을까?

이벤트는 Redux의 Action처럼 식별자와 페이로드를 가지고 있는 객체이다. 그리고 `album_list`와 `dialog_to_add_photo`는 같은 이벤트를 낚아챌 수 있어야 한다. 이벤트가 스스로를 구독하는 특정한 객체를 알아야 한다면 확장성은 떨어질 것이다. 따라서 발생한 이벤트를 어디서나 접근할 수 있도록 만드는 편이 낫다.

> 설계 원칙 2. 발생한 이벤트는 전역적으로 접근할 수 있다.

그전에 이벤트는 어디서 발생할까? Redux에서 Action를 발생시키는 주체는 두 가지다.
1. **컴포넌트가 Action을 발생시킨다.** 컴포넌트는 주로 이용자의 입력에 대한 이벤트를 발생시킨다. 버튼을 눌렀을 때나 입력란을 수정했을 때와 같은 경우들이다. 또는 컴포넌트가 생성되거나 파괴되었을 때도 이벤트는 발생할 수 있다.
```jsx
const App = ({ dispatch }) => {
  return (
    <button onClick={() => dispatch(action())}>
      확인
    </button>
  )
}
```
2. **미들웨어가 Action을 발생시킨다.** 미들웨어는 각종 사이드이펙트를 일으킬 수 있다. 꼭 이용자에 의해서가 아니더라도 프로그램에는 입력이 들어온다. 비동기로 예약한 HTTP 요청, 연결한 웹소켓에서 들어오는 메시지 등등이 있다.
```js
const fetchTodosMiddleware = (storeApi) => (next) => (action) => {
  if (action.type === 'fetch-todos') {
    client.get('todos').then(todos => storeApi.dispatch({
      type: 'todos-loaded',
      payload: todos,
    }))
  }

  next(action)
}
```

보여준 자바스크립트 코드는 함수형이다. Flutter 앱에 적용한다면 어떨까? 이른바 Compiler Driven Development를 실천해 보자.

```dart
class SignInForm extends StatelessWidget {
  @override
  Widget build(BuildContext context) {
    return Scaffold(
      body: Colum(
        children: [
          ...
          TextButton(
            onPressed: () => dispatch(SubmitEvent(...)),
            child: Text("제출")
          )
        ],
      )
    );
  }
}
```

`dispatch()` 함수를 사용하는 방법은 여러가지일 것이다. 전역에 선언할 수도 있다. 하지만 위험한 선택이다. 이벤트를 발생시키는 주체는 한정되어 있다.

> 설계 원칙 3. 이벤트를 발행하는 주체는 한정되어야 한다.

떠오르는 방법은 Widget을 상속하는 부모 클래스에 `dispatch()` 메서드를 선언하는 것이다. Flutter에서 Widget을 만들기 위해 StatelessWidget을 상속해야 하는 것처럼 말이다.

```dart
abstract class SuperWidget extends StatelessWidget {
  @protected
  void dispatch(Event event) {
    ...
  }
}
```

다시 돌아와서 `dispatch()`는 인자로 전달받은 `event`를 전역에 노출시켜야 한다. 하지만 이때 전역이라는 말은 모든 객체를 의미하지 않는다. `event`를 발행하는 주체처럼 사용하는 주체도 한정되어 있다. 다만 트리 구조에도 불구하고 어디서나 접근할 수 있어야 한다는 의미다.

> 설계 원칙 4. 이벤트를 사용하는 주체 역시 한정되어야 한다.

Store는 이벤트를 수신한다. Redux에서는 Reducer와 Middleware의 인자로 전달할 Action을 Store가 받아 오는 식이다. Redux Store의 일반적인 상태 변화 메커니즘을 모사하면 다음과 같이 아주 간단하다.

```dart
typedef Reducer<T> = T Function(T, Event);

T reducer<T>(T state, Event event) {
  switch (event.type) {
    case "do-something":
      return newState;
    default:
      return state;
  }
}

class Store<T> {
  T _state;

  final Reducer<T> reducer;

  Store(this.reducer);

  void dispatch(Event event) {
    _state = reducer(_state, event);
  }
}
```

하지만 이대로 Dart와 같은 정적 언어로 옮겨 오면 문제가 있다. 우선 차후에 컴포넌트 측에서 가져올 Store를 특정하지 못한다. Redux는 문자열을 적극적으로 이용하고 있다. 하지만 그건 Redux Store의 상태가 자바스크립트 오브젝트이기 때문에 가능한 일이다. Dart도 자바스크립트 오브젝트와 유사한 Map이 있지만 타입이 지워지기 때문에 정적 언어의 혜택을 누리지 못한다. 당연하게도 타입스크립트와 같은 타입 체계도 아니다.

그러므로 개발자의 입장에서 Store를 아래처럼 작성한다면 어떨까?
```dart
class ListOfAlbumStore extends Store<List<AlbumModel>> {
  @override
  List<AlbumModel> reducer(List<AlbumModel> state, Event event) {
    if (event is ListOfAlbumLoaded) {
      return [...state, ...event.payload]
    }

    return state;
  }
}
```

다음과 같은 스타일도 가능하다.

```dart
listOfAlbumLoadedReducer(List<AlbumModel> state, ListOfAlbumLoaded event) {
  return [...state, ...event.payload];
}

class ListOfAlbumStore extends Store<List<AlbumModel>> {
  ListOfAlbumStore(
    reducers: {
      ListOfAlbumLoaded: listOfAlbumLoadedReducer,
    },
  );
}
```

핵심은 `ListOfAlbumStore`과 같은 클래스의 타입을 통해 Store를 특정할 수 있게 되었다는 점이다. 이어지는 API를 사용하면 데이터의 타입도 알 수 있다.

```dart
T get<T>();
```

실제로는 다음과 같이 사용된다.

```dart
class AlbumList extends StatelessWidget {
  @override
  Widget build(BuildContext context) {
    return Scaffold(
      body: StreamBuilder(
        stream: get<AlbumListStore>().stream,
        builder: ...
      )
    );
  }
}
```

해당하는 메서드는 Widget 외에는 사용할 수 없다. 따라서 위젯은 이제 `void dispatch(Event)`, `T get<T>()` 두 API를 구현해야 한다.

```dart
abstract class SuperWidget extends StatelessWidget {
  @protected
  T get<T>() {
    return ...
  }

  @protected
  void dispatch(Event event) {
    ...
  }
}
```

## 사이드이펙트 관리
지금까지는 의도적으로 Middleware를 도외시했다. Widget뿐만 아니라 Middleware도 `void dispatch(Event)` 메서드를 사용한다. 또한 Store뿐만 아니라 Middleware도 Action을 수신한다. 하지만 Redux Middleware에 대해 남아 있는 의문은 있다.

```js
type middleware = (storeApi) => (next) => (action) => void
```

Middleware도 Action에 의해 촉발된다. Reducer와 같다. 그러나 근본적인 차이가 있다. Reducer의 핵심은 상태를 반환한다는 것이다. 반면 Middleware는 상태를 반환하지 않는다. Reducer와는 달리 마지막 부분에 `next()`메서드를 사용한다. 하지만 나머지 Middleware와 Reducer를 가로막지 않기 위한 목적일 뿐이다.

인자로 전해받은 storeApi에서 Middleware는 주로 `storeApi.getState()` 혹은 `storeApi.dispatch()` 메서드를 호출한다. Store의 상태를 읽거나 새로운 Action을 발행하는 것이다. 이러한 행동은 차라리 위에서 본 Widget과 닮아 있지 않은가? Widget에 구현한 `T get<T>()` `void dispatch(Event)` 메서드를 떠올려 보자.

Redux는 꼭대기에 자리하는 하나의 Store를 통해 중앙화를 꾀한다. 그러므로 Middleware를 Store에 등록시키는 방법은 물론 상식적이다. Store는 액션을 받는 유일한 통로이므로 Reducer 이전에 등록해 두면 Middleware를 실행시킬 수 있다.

```js
const dispatch = middlewares.reduce(
  (acc, cur) => cur(this)(acc),
  (action) => {
    currentState = reducer(currentState, action)
  },
)

dispatch(action)
```

하지만 지금 우리는 여러 개의 Store를 가지고 있다. 나아가서 여러 개의 Store에 이벤트를 조달하는 메커니즘 역시 가지고 있을 것이다. 그렇다면 Middleware에게 이벤트를 조달하는 것 역시 어렵지 않다.

Middleware는 간혹 상태를 읽어야 하지만 특정한 Store와는 관련이 없다. 또한 이벤트를 발행해야 하지만 이벤트는 전역으로 퍼져 나간다. 기왕의 `storeApi`가 제공하는 메서드를 쓸 일이 없어진다. 위에서 Widget에 구현한 것과 동일한 메서드를 가지는 편이 바람직하다.

따라서 Middleware는 아래와 같은 형태가 된다.

```js
type Middleware = (get, dispatch, action) => void
```

`next()` 메서드를 사용하지 못하니 더 이상 Middleware라고도 할 수 없다. Effect라고 이름지어 보자. 함수형도 상관 없지만 클래스로 만들면 다음과 같을 것이다.

```dart
abstract class Effect {
  @protected
  T get<T>() {
    return ...
  }

  @protected
  void dispatch(Event event) {
    ...
  }

  @protected
  void execute(Event event);
}
```

Effect의 구체 클래스를 작성하는 개발자는 의무적으로 `void execute(Event)` 메서드를 재정의한다. 와중에 상태를 읽고 싶다면 `T get<T>()`를 사용하고 이벤트를 발행하고 싶다면 `void dispatch(Event)` 메서드를 호출하는 것이다.

`void execute(Event)` 메서드 내부에서는 모든 종류의 사이드이펙트가 허용된다. Store의 상태를 변화시키는 데에 관여할 수도 있다. 핵심적인 역할은 Reducer에게 이벤트가 도달하기 전에 모든 사이드이펙트를 처리하는 것이다. Redux의 Middleware와 구조는 다르지만 여전히 중간 다리 역할을 해낼 수 있다.

> 설계 원칙 5. Effect에서 모든 사이드이펙트를 도맡아 관리한다.

Middleware는 Store에 등록된다. 정확히 말해서는 Store의 Reducer를 강화시킨다. Effect는 어디에 등록되어야 할까? 이대로라면 어디에다 두어도 좋을 성싶다. Widget이 트리 구조 내에 어디에나 자리할 수 있는 것처럼 말이다.

## 생명주기
오늘날 프론트엔드 프레임워크의 핵심 컴포넌트는 생명주기를 가지고 있다. 애플리케이션은 생성되었다 소멸된다. 하나의 화면도 마찬가지다. 화면을 구성하는 하나하나의 컴포넌트들도 상황에 따라 생성되었다 소멸된다.

어떤 서비스의 회원가입 절차에 관해서 생각해 보자. 첫 번째 화면에서는 우선 인증을 거친다. 두 번째 화면에서는 서비스 이용에 관한 동의를 받는다. 세 번째 화면에서는 기본적인 회원 정보를 제출하게끔 한다.

세 개의 화면을 사용하지만 상태는 하나로 둘 수 있다면 유용하다. 이를테면 `SignUpStore`라고 하자. 우리의 Store는 Redux의 Store와는 달리 전역적이지 않으므로 생명주기를 고려해야 한다. `SignUpStore`는 회원가입 절차 중 첫 번째 화면에서 생성되고 마지막 화면이 종료될 때 함께 소멸되어야 하는 것이다.

물론 애플리케이션 전체 주기 동안 유지되는 Store도 있다. 혹은 하나의 화면이나 하나의 컴포넌트와 함께 생성되었다 소멸되는 Store도 있다. `SignUpStore`는 그보다 애매하기 때문에 구체적인 요구 사항을 파악하는 데에 도움이 된다.

`SignUpStore`를 어떻게 생성해야 하며 소멸시켜야 할까? 보통은 컴포넌트의 생명주기에 맞물리면 된다. 이때는 컴포넌트가 생성되고 소멸하는 시점을 결정한다.

```ts
class Component {
  private readonly storeConstructors: StoreConstructor[];

  private stores: Store[];

  public constructor({
    storeConstructors,
  }: {
    storeConstructors: StoreConstructor[]
  }) {
    this.storeConstructors = storeConstructors;
  }

  protected componentWillMount(): void {
    this.stores = this.storeConstructors.map(({ construct }) => construct());
  }

  protected componentWillUnmount(): void {
    this.stores.foreach(({ destroy }) => destroy());
  }
}
```

애플리케이션 범위에도 사용할 수 있고 컴포넌트 범위에서도 사용할 수 있다. 하지만 몇 개의 화면을 아우르는 범위에서 사용하기는 어렵다. 특히나 웹 프레임워크들은 특정한 다수의 화면을 포함하는 wrapper를 거의 지원하지 않는다. 이러한 상황에서는 오히려 Store가 자신의 생성 시점과 소멸 시점을 결정할 수 있게 하는 편이 좋을 것도 같다.

```dart
class SignUpStore extends Store<SignUpModel> {
  SignUpStore(
    reducers: {
      FormPending: formPendingReducer,
      FormSubmitted: formSubmittedReducer,
      ...
    },
    constructWith: SignUpAuthentication,
    destoryWith: SignUpFillInUserInfo,
  );
}
```

혹은 생명주기를 결정하는 권한을 다시 컴포넌트에게 넘기는 방법이 있다.

```dart
class SignUpAuthentication extends Component {
  SignUpAuthentication() : super(
    stores: [
      Singleton(
        () => SignUpStore(),
        constructWith: SignUpAuthentication,
        destroyWith: SignUpFillInUserInfo,
      ),
    ],
  )

  @override
  Widget build(BuildContext context) {
    return ...
  }
}
```

`SignUpStore`는 `SignUpAuthentication` 노드와 함께 생성되고 `SignUpFillInuserInfo` 노드와 함께 소멸하는 싱글톤 객체다. 이때 컴포넌트인 노드는 생명주기의 핵이기 때문에 직관적인 것처럼 보인다. Effect도 마찬가지로 노드 단위로 등록시키면 좋을 것이다.

```dart
class SignUpAuthentication extends Component {
  SignUpAuthentication() : super(
    stores: [
      Singleton(
        () => SignUpStore(),
        constructWith: SignUpAuthentication,
        destroyWith: SignUpFillInUserInfo,
      ),
    ],
    effects: [
      Factory(
        () => SubmitEffect(),
        constructWith: SignUpAuthentication,
        destroyWith: SignUpFillInUserInfo,
      ),
    ],
  )

  @override
  Widget build(BuildContext context) {
    return ...
  }
}
```

Effect는 특정한 맥락을 가진 노드의 생명주기와 맞물린다. 따라서 Redux의 Middleware와는 다르게 항상 실행되지 않는다.

> 설계 원칙 6. 노드는 Store와 Effect의 생명주기를 관리한다.
