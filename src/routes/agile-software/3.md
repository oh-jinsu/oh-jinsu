지난 세 차례의 프로젝트에서 나는 MVVM 아키텍처를 채택했다. MVVM은 매력적인 아키텍처다. 다음과 같은 분명한 장점들이 있다.
1. **구조가 단순하다.** MVVM 아키텍처는 소프트웨어를 최소한도의 계층으로 분리한다. 이용자가 당장 마주하는 뷰(View)와 그것에 대응하는 정보의 현상태로서 뷰모델(ViewModel)이 있으며, 인터페이스를 통해 가려지는 정보의 원상태(Model)가 있다. 개발자는 개념에 맞게 코드를 분산시켜 소프트웨어의 유지보수를 보다 간편하게 만들 수 있다. 이는 계층의 개념적 구분이 비교적 명료한 까닭에 그다지 어려운 일이 아니다.
2. **가장 분리되어 있다.** MVC. MVP. 그리고 MVVM. 보이는 것처럼 모델과 뷰의 관계보다는 뷰와 컨트롤러, 프레젠터, 혹은 뷰모델과 같은 개념들 사이의 관계가 중요하다. MVC 뷰와 모델 사이의 협착이 있다. MVP에서 뷰모델은 자신의 변화를 알려야 하며, 뷰는 변화를 수용할 메서드를 추가적으로 구현해야 한다. 역시나 의존 관계가 있다. 그러한 의존 관계를 MVVM은 흔히 이야기하는 데이터 바인딩을 통해 해결한다. 핵심은 뷰모델이 정보를 뷰의 관점에서 있는 그대로 이용할 수 있는 형태로 저장하고, 뷰는 정보를 구독하는 것이다.
3. **테스트에 최적화되어 있다.** MVVM 아키텍처의 진가가 여기에 있다. 변화하는 부분은 검증되어야 한다. 무언가를 테스트하기 위해선 우선 대상을 정보의 형태로 가공해야 한다. 따라서 MVVM은 뷰에 대한 모델인 뷰모델을 만든다. 뷰모델이 이용자가 마주하는 뷰에 대한 근사치를 표현하므로 개발자는 까다로운 뷰의 테스트에서 해방될 수 있다. 대부분은 뷰모델만 테스트하면 된다. 나아가 순전히 정보의 올바른 변화만을 관찰하기 위해 기반 시스템의 구현부를 외부 의존성으로 추출한다. 의존성들을 가짜 객체로 치환하면 소프트웨어를 실제로 구동하지 않고서도 여러가지 상황을 빠르게 시뮬레이션할 수 있다.

이처럼 분명한 장점들에도 불구하고 직접 구현하면서 느낀 단점들도 많다. 때때로 위의 특성들 자체가 한계를 보여주기도 한다. 글을 적는 목적은 보다 나은 설계를 위해 우선 문제점을 분명히 인식하고 해결의 실마리를 이끌어 내기 위해서이다.

## MVVM 아키텍처는 사실 단순하지 않다.
### 더 구체적으로 생각해 보기 1: 하나의 뷰에 여러 개의 뷰모델이 필요한 경우
일반적으로는 모바일 애플리케이션과 궁합이 좋다. 하나의 화면을 하나의 뷰로서 취급한다면, 하나의 화면마다 하나의 뷰모델을 둘 수 있다. 그러면 소프트웨어의 구조는 분명히 단순해진다.
```
+-- screens
|   +-- home
|       +-- viewmodel
|       +-- view
|   +-- signin
|       +-- viewmodel
|       +-- view
|   +-- profile
|       +-- viewmodel
|       +-- view
.
.
.
```
한 화면에 하나의 기능. 이 세련된 원칙을 지킬 수 있다면 위의 구조는 이상적이다. 모바일 애플리케이션에서는 이러한 일이 많다. 하지만 보다 넓은 지면을 활용하는 웹에서는 그렇지 않다. 상이한 성격을 가진 여러 요소들이 하나의 화면 위에 공존한다. 물론 모바일 애플리케이션도 아주 예외는 아니다. 홈 화면에는 종종 다종다양한 컴포넌트들이 끼어든다. 

멀리 가지 않더라도 하단 네비게이션을 가지고 있는 익숙한 화면들의 경우가 그렇다. 하단 네비게이션 뷰에게도 현재 상태를 나타낼 모델은 필요하다. 이를테면 현재 위치를 가리키는 색인 값이 그것이다. 색인 값이 없다면 현재 위치를 나타내는 아이콘을 하이라이트할 수 없을 것이다. 나아가서 채팅 등의 새로운 피드를 받는 애플리케이션이라면 읽지 않은 피드의 개수 또한 가지고 있어야 한다.

이때 역시 화면 단위로 뷰모델을 구성한다면 어떤 일이 생길까? 가능하지만 중복이 생긴다. MVVM 아키텍처에서 뷰모델은 비즈니스 로직을 포함하는 메서드를 함께 구현하게 마련이다. 그러한 관례를 따르자면 우선 하단 네비게이션의 아이콘을 클릭했을 때 작동할 코드를 화면마다 다시 작성하는 수고를 감내해야 한다. 예를 든 것처럼 피드를 받아야 한다면 어떨까? 매 화면의 뷰모델마다 다시 처음 상태를 불러오고, 계속 현재 상태를 갱신할 수 있도록 다시 웹소켓을 연결해 주어야 할 것이다. 이건 단순히 개발자를 고생시키는 의미를 넘어선다. 즉 네트워크 자원을 낭비하게 되는 것이다.

이렇게 되면 조금의 재치를 발휘해서 하단 네비게이션과 같이 공통된 컴포넌트를 독립적인 단위로 취급하고 싶어진다. 아마 다음과 같이 구조를 잡으면 될지도 모른다.

```
+-- components
|   +-- bottom_navigation
|   |   +-- viewmodel
|   |   +-- view
|   +-- about
|   |   +-- viewmodel
|   |   +-- view
|   +-- signin_form
|   .   +-- viewmodel
|   .   +-- view
|   .
```
일단 최상위의 범주를 components라고 변경해야 했다. 하단 네비게이션이 화면은 아니니까 말이다. 기왕의 화면들도 컴포넌트의 개념에 맞게 재정의했다. 디렉토리 구조는 여전히 깔끔하다. 하지만 이렇게 추출된 하단 네비게이션 컴포넌트는 어떻게 다시 삽입되어야 할까? 두 가지 방법이 떠오른다.

```dart
class HomeView extends StatelessWidget {
  final viewModel = Dependency.inject<HomeViewModel>();

  @override
  Widget build(BuildContext context) {
    return Scaffold(
      body: ..., // TODO: Use the HomeViewModel
      bottomNavigationBar: BottomNavigationView(),
    );
  }
}
```
UI를 그리기 위해 선언적 구문을 사용하는 Flutter 스타일로 작성하였지만 예컨대 안드로이드도 마찬가지다. 디렉토리 구조상으로 동일한 층위에 나란히 있는 두 컴포넌트가 실은 중첩된(Nested) 관계를 맺는다. `HomeView`는 `BottomNavigationView`에 의존하는 것처럼 보인다. 다른 방법도 있다.
```jsx
const HomePage = () => {
  return (
    <div>
      <header>
        <TitleComponent />
        <NavigationComponent />
      </header>
      <main>
        <AboutComponent />
      </main>
      <aside>
        <AdComponent />
      </aside>
      <footer>
        <FooterComponent />
      </footer>
    </div>
  )
}
```
조금 더 복잡한 화면의 예시를 들었다. Component를 Flat하게 나열할 수 있는 Wrapper가 필요한 것이다. 물론 이 Wrapper는 또한 가시성을 가지고 있는 View라고 말할 수도 있다. 일반적으로 컴포넌트들이 아무렇게나 배치되어 있지는 않기 때문이다. 하지만 ViewModel을 가지고 있지 않은 View라면 MVVM에서 큰 의미를 가지지는 않는다. 독립적인 요소라고 생각해도 좋을 것이다.
```
+-- app
|    +-- pages
|    |   +-- home
|    |   +-- signin
|    |   .
|    |   .
|    |   .
|    +-- components
|        +-- bottom_navigation
|        |   +-- viewmodel
|        |   +-- view
|        +-- about
|        .   +-- viewmodel
|        .   +-- view
|        .
```
디렉토리 구조는 아마도 이와 비슷한 모습이 된다. 나름 정리가 된 것처럼 보인다.

### 더 구체적으로 생각해 보기 2: 여러 개의 뷰에 하나의 뷰모델이 필요한 경우

지금까지는 한 화면에서 여러 개의 뷰모델이 필요한 일반적인 경우에 대해 고려했다. 또 다른 일반적인 경우가 있다. 여러 개의 화면을 넘나드는 와중에 하나의 뷰모델이 필요한 경우들이다. 구글에서 친절하게 마련한 안드로이드 튜토리얼에서는 컵케이크를 주문하는 플로우를 예시로 들고 있다.

한 화면에 하나의 기능. 하지만 기능의 구획은 매번 임의적일 수밖에 없다. 컵케이크를 주문하기. 이건 하나의 기능이다. 하지만 그러한 과정 속에 있는 가령 주문 종류와 수량 입력하기, 배송지 입력하기 등도 역시 각각 하나의 기능으로 여겨질 수 있다. 하지만 그처럼 작은 단위마다 뷰와 뷰모델을 구현하면 문제가 발생한다. 뷰모델은 다른 뷰모델을 의존할 수 없으므로 상태가 공유되지 않는 것이다.

이러한 문제를 해결하기 위해 구글의 안드로이드 튜토리얼에서는 SharedViewModel이라는 개념을 제안한다. 뷰에 대해 1:n의 관계를 가지는 뷰모델이다. 이 뷰모델은 뷰보다 긴 생명주기를 가질 수 있다. 직관적이고 간단한 해결책이다. 그런데 소프트웨어의 구조를 나타내는 디렉토리 구조상에서는 어디에 위치해야 하는 걸까?
```
+-- app
|    +-- pages
|    |   +-- home
|    |   +-- order1
|    |   +-- order2
|    |   .
|    |   .
|    |   .
|    +-- components
|        +-- bottom_navigation
|        |   +-- viewmodel
|        |   +-- view
|        +-- about
|        |   +-- viewmodel
|        |   +-- view
|        +-- order
|        .   +-- viewmodel
|        .   +-- views
|        .       +-- view1
|                +-- view2
|                .
|                .
|                .
```
바로 뷰들의 위치가 이동한다. 적힌 order는 더 이상 컴포넌트라고 부르기 애매할지도 모른다. 그도 그럴 것이 order는 차라리 컴포넌트들의 집합일 것이기 때문이다. 적어도 가져오는 pages의 입장에서 order는 여러가지 형태를 띤다. 하지만 이름 정도야 큰 문제는 아니다.

SharedViewModel은 문제를 아주 쉽게 해결하는 것처럼 보인다. 하나의 뷰모델이 여러 개의 View를 감싸는 방법을 통해서 말이다. 하지만 그만큼 뷰모델은 비대해진다. 그럴 수밖에 없는 것은 재차 말하지만 뷰모델은 테스트 가능하기 위해서 어떤 것도 의존하지 않아야 하기 때문이다.

심각하게 체감하기 위해서 아주 일상적으로 경험하는 이용자 정보에 대한 예를 들어 보자. 물론 프로필 화면으로 이동하면 이용자의 정보를 전체적으로 보여주겠지만, 종종 상단 네비게이션 바의 우측에도 조그마한 프로필 아바타가 위치한다.

일반적으로 정보의 원천은 같다. Restful API를 구현한 백엔드와 통신한다면 GET /users/:id를 통해 이용자 정보를 불러올 필요가 있다. 네비게이션 바와 프로필 화면이 서로 다른 컴포넌트인 까닭에 독립적인 뷰모델을 가지고 있다면 당연히 통신은 두 번 일어나야 한다. 곱절로 소모되는 네트워크 자원이 아까워서 이번에도 두 컴포넌트를 감싸는 거대한 뷰모델을 만들면 만사가 쉽게 해결될까? 결코 그렇지 않다. 마지막엔 뷰모델에 애플리케이션 전체를 아우르는 비즈니스로직을 몽땅 밀어넣고 나서야 끝날 것이다.

어떻게 할 것인가? 그동안 나는 아키텍처를 무너뜨리는 편보다는 네트워크 비용을 감수하는 쪽을 택했다.

## 알다시피 뷰모델은 과도한 책임을 떠안는다.
이른바 Massive View Controller는 여기에도 있다. 근본적으로 달라진 것이 딱히 없는 탓이다. 방금 살펴 본 SharedViewModel만 참고하더라도 알 수 있다. 뷰모델은 당초에 뷰에 대한 모델이었다. 핵심은 뷰가 뷰모델의 데이터를 있는 그대로 획득(get)할 수 있어야만 한다는 것이다. 하지만 SharedViewModel은 이미 뷰모델을 상태를 저장하는 용도까지 확장하고 있다. 예를 들어 안드로이드 Kotlin 코드를 보자.

```kt
class ProfileViewModel: ViewModel {
  private val user = MutableLiveData<UserModel>()

  val username: LiveData<String> get() = MediatorLiveData<String>().apply {
    addSource(user) { value = it.username }
  }

  val avatarImageUri: LiveData<String> get() = MediatorLiveData<String>().apply {
    addSource(user) { value = it.avatarImageUri }
  }
}
```

물론 이러한 형태가 아주 간편하다. 뷰가 있는 그대로 받아야 할 username과 avatarImageUri 획득자 속성은 다시금 user라는 상태를 구독한다. 주어진 예처럼 뷰모델은 그러한 현재 상태를 여러가지 형태의 획득자 속성을 위해 미리 준비할 수도 있지만, 다시 스스로 상태를 사용할 수도 있다. 그리고 그건 아주 흔한 일이다. 왜냐하면 일반적으로 뷰모델은 이용자에게 출력할 획득자 속성뿐만이 아니라 이용자의 입력을 수신하는 메서드도 함께 가지고 있기 때문이다.

```kt
class SignInFormViewModel(
  private val authClient: AuthClient,
): ViewModel {
  private val _email: MutableLiveData<String>("")

  val email: LiveData<String> get() = _email;

  private val _password_: MutableLiveData<String>("")

  val password: LiveData<String> get() = _password;

  fun onEmailChanged(value: String) {
    _email.value =  value;
  }

  fun onPasswordChanged(value: String) {
    _password.value = value;
  }

  fun onSubmitted() {
    val email = _email.value;
    val password = _password.value;

    val result = authClient.signIn(email, password)

    ...
  }
}
```
아마도 이용자가 이메일과 비밀번호를 입력한 뒤에 제출버튼을 누르면 `onSumibtted()` 메서드가 실행될 것이다. 이때 메서드는 앞서 저장된 상태를 사용한다. 짚고 넘어가자면 인자를 통해 이메일과 비밀번호를 받지 않는 이유가 있다.
1. **메서드를 호출하는 뷰가 인자를 전달하기 어렵다.** HTML에는 Form Element가 있어서 입력된 값들을 제출한 시점에 모두 읽어들일 수 있다. 하지만 모든 프레임워크에 존재하는 일반적인 기능은 아니다. 뷰는 무상태성을 가져야 하며 일종의 순수 함수여야 한다. 다시 말해서 나중에 전달할 상태를 저장하고 있어서는 안 된다.
2. **뷰로 내보낸 데이터는 모델 단에서 이용하는 형태와 차이가 있을 수 있다.** 모델을 굳이 뷰모델의 획득자 속성으로 변환하는 이유이기도 하다. 이를테면 패스워드는 `*`와 같이 특정한 기호로 은닉되어 내보내야 할 수 있다. 또한 `10000`처럼 정수로 존재하는 금액은 `"10,000"`와 같이 적절한 포맷으로 내보내야 한다. 즉 뷰는 데이터의 원형을 알지 못한다. 그러므로 뷰모델에게 유용한 인자를 제공하기 어렵다.

위의 코드를 자세히 보자. 뷰모델은 적어도 네 가지 책임을 지니고 있다.
1. 획득자 속성을 구현한다.
2. 현재 상태를 저장한다.
3. 이벤트에 대한 입력을 받는다.
4. 입력에 따라 비즈니스 로직을 처리한다.

사람들은 뷰모델에서 비즈니스 로직을 관리한다고들 이야기한다("상태 관리"라고도 하지 않던가). 물론 아키텍처를 아주 사용하지 않을 때보다는 훨씬 낫다. 하지만 뷰모델은 여전히 온갖 일들을 다 처리하고 있다. 내가 보기에는 단순하게 뷰와 레포지토리를 분리해 놓고서는 어떠한 체계도 없이 코드를 작성하는 것처럼 보인다. "비즈니스 로직"이란 무언가에 대한 분명한 개념이 아니라 정리되고 남은 나머지 모든 복잡한 코드를 뭉뚱그릴 뿐인 셈이다.

이 복잡한 코드를 관리하기 위해 나는 어떻게 했는가? 테스트 코드를 작성했다.

## 지겨운 테스트 코드가 강제된다.
예쁜 말로는 Test Driven Design이라고 한다. 나도 한때는 이 개념에 매료되었었다. 잘 작성한 테스트 코드는 확실히 쿨하다. 실제 소프트웨어를 구동하여 수행하는 테스트와는 비교되지 않을 정도로 빠르게 특정한 상황을 시뮬레이션할 수 있다.

그리고 MVVM 아키텍처를 채택한다면 나는 여전히 테스트 코드를 작성할 것이다. 왜냐하면 앞서 보았듯이 MVVM 아키텍처의 아주 복잡한 상태를 관리하고자 한다면 수많은 테스트를 거치는 방법밖에는 없기 때문이다.

이건 처음에는 짜릿하지만 겪다 보면 일반적으로 매우 지겨운 과정이다. 일단 테스트를 위해 프레임워크마다 제각각인 초기 세팅에 시간을 들여야 한다. 그 다음에는 적든 많든 보일러플레이트 코드를 매번 작성하고 있다. 뷰모델의 의존성을 가짜 객체로 치환하고 특정한 테스트 케이스에 맞는 반환값을 선언해 주는 식이다. 그리고 커버리지 100%를 향해 모든 함수를 테스트한다. 다행히도 완벽하게 자명한 코드는 존재하지 않는다.

다시 말하지만 완벽하게 자명한 코드는 존재하지 않는다. 무슨 말인가? 이건 바로 "지겨운 테스트 코드"가 단순한 투정이 아닌 이유이다. 익숙한 말로는 다음과 같다.

> 테스팅은 결함의 존재를 보여줄 수는 있다. 하지만 결함의 부재를 증명할 수는 없다.

한번이라도 테스트 코드를 작성해 봤다면 잘 알 것이다. 모든 테스트 코드가 통과한 다음에도 버그는 있다. 이건 기껏해야 뷰모델의 복잡성이 사람들의 예측을 뛰어넘을 정도로 복잡하거나 그러므로 복잡성을 모두 타진하기에는 현실적인 한계가 있기 때문이 아니다. MVVM 아키텍처와 그것의 테스팅이 기반하고 있는 객체 지향 프로그래밍이 원리적으로 허점을 노출하는 탓이다.

우리가 그동안 작성한 테스트 코드를 돌이켜 보자. 어떤 식이었는가? 앞서 예를 든 뷰모델을 활용하자.
```kt
val authClient = MockAuthClient()

val viewModel = SignInFormViewModel(
  authClient,
)

viewModel.onEmailChanged("an email")

assertEquals("an email", viewModel.email.getOrAwaitValue())
```
주어진 코드에서 우리는 `email`이라는 획득자 속성이 `"an email"` 문자열과 동등하다고 단언한다. 우선 `email` 속성이 변화를 잘 반영하는지를 관찰하는 것이다. 왜냐하면 이것이 우리가 최종적으로 제공하려는 값이기 때문이다. 그것에 의해 `onEmailChanged()` 메서드가 올바르게 작동하는지가 유추된다.

```cs
class SignInFormViewModel {
  public string email;

  SignInFormViewModel() {
    this.email = "";
  }

  public void ChangeEmail(string value) {
    this.email = value;
  }
}
```

새삼스럽지만 `email`은 변수다. 즉 바뀔 수 있는 값이다. 변화에 대해 항상 열려 있다. 위의 C# 코드에서 `email`의 현재 상태를 알기 어려운 이유는 뭘까? 주어진 `ChangeEmail()`함수처럼 `email`이라는 변수를 직접 조작하는 함수가 어딘가 또 있을지도 모르기 때문이다. `email` 속성이 전역에 노출되어 있는지 은닉되어 있는지가 중요한 게 아니다. 클래스 내부에도 `email`을 마음대로 조작할 수 있는 많은 코드들이 있다.

```cs
class Foo {
  public int a;

  Foo() {
    this.a = 0;
  }

  public void Add(int b) {
    this.a + b;
  }
}
```

따라서 더욱 단순하고 일반적인 위의 코드조차 의심하지 않을 수 없다. 주어진 함수 `Add()`를 보는 것처럼 객체지향프로그래밍은 `void`라는 반환값을 허용한다. 그런데 중학생 때 우리는 함수라면 y=f(x)처럼 특정한 입력값에는 항상 그에 대응하는 결과값을 가져야 한다고 배우지 않았는가? 하지만 `Add()`는 아무것도 반환하지 않는다. 아무것도 반환하지 않는다면 함수는 도대체 무슨 의미가 있을까? 물론 객체지향프로그래밍에서는 의미가 있다. `Add()`는 a를 조작하는 사이드이펙트를 가지고 있기 때문이다. 그러한 사이드이펙트마저 없다면 `Add()`는 아무짝에도 쓸모없을 코드에 불과할 것이다.

따라서 객체지향프로그래밍의 메서드와 "순수함수"는 그 결과값을 도출해내는 근본적인 방법에서 차이를 보인다. 객체지향프로그래밍에서는 특정한 입력 이후에 변수가 가지는 일시적인 상태를 확인한다.
```cs
Foo foo = Foo();

foo.add(1);

Console.Writeline("{0}", foo.a);
```

순수함수는 특정한 입력에 대해 영구적으로 동일한 결과값을 반환한다. 외부의 변수를 참조하는 일이 없다. 나아가서 함수형 프로그래밍은 변수를 사용하지 않는다.
```js
add(a, b) {
  return a + b
}

const result = add(0 + 1)

console.log(result)
```

위의 자바스크립트 코드에서 `result`의 값은 테스트되어야 할까? 물론 한 번쯤 확인할 필요는 있겠지만 거창한 테스트코드를 작성할 필요까지는 없을 것이다. 왜냐하면 우리가 확인하고자 하는 결과 값 `result`는 키워드가 보여주는 것처럼 불변이기 때문이다.

하지만 먼저 예시한 C#코드에서 `foo.a`는 어떠한 시점에 테스트를 통과한다고 하더라도 무결성을 영원히 보장할 수 없다. 왜냐하면 그건 애당초 변수이지 않은가.

이것이 MVVM 아키텍처에서 우리가 테스트를 멈추지 말아야 할 근본적인 이유이다. 뷰로 내보내야 할 결과가 다름 아닌 뷰모델이 들고 있는 변수이기 때문이며, 이 변수는 스스로가 알지 못하는 무언가에 의해 언제나 조작될 수 있기 때문이다.

테스트 코드를 작성하는 개발자들은 변수가 아주 특정한 시점에는 결함이 없음을 증명한다. 테스팅은 그러한 식으로 무한한 시간을 채워나가야 하는 지난한 과정이다.

## 더 나은 소프트웨어 아키텍처를 향해
경험한 MVVM의 문제점들에 대해 불평불만을 늘어놓았다.
### **일반적인 요구사항이지만 MVVM을 적용하기 어려운 경우들이 많다.**

단순히 연습용으로 제품을 만들지 않는다면 빈번한 일이다. 고민에 빠지게 마련이다. 아키텍처를 유지하기 위해서는 반복 작업에 드는 수고나 네트워킹 비용 등을 감수해야 한다. 순간적인 필요에 따른 무분별한 개념의 확장 역시 아키텍처를 무너뜨릴 위험을 안고 있다.

현대 애플리케이션의 일반적인 구조에 적합한 모형을 실제 구현한 경험 속에서 탐색하는 일이 요구된다. 컴포넌트 단위로 뷰모델을 쪼개는 시도는 좋아 보인다. 문제는 중첩되거나 포함하는 구조를 풀어 나가는 방법이다. 기왕의 뷰모델보다 더욱 작고 유연한 단위가 필요한 것이다.

### **MVVM의 핵심 계층인 뷰모델은 여전히 책임이 덜 분리되어 있다.**

뷰모델은 적어도 네 가지 일을 동시에 수행한다. 입력과 출력을 관리하고, 현재 상태를 저장하며 변화를 산출하는 것이다. 뷰모델은 쉽사리 비만 상태에 빠진다. 그리고 복잡한 하나의 뷰모델은 그만큼 끔찍한 테스트 코드를 강제한다.

흔히들 쉽게 이야기하는 "비즈니스 로직"들은 더욱 정교한 수준으로 분리될 필요가 있다. 뷰모델을 다시 일종의 데이터 구조체로 축소하는 것은 어떨까. 그리고 그와는 다른 곳에 가능한 입력을 나열해 보는 것은 어떨까. 상태를 저장하는 객체를 따로 위치시키고, 상태의 변화는 또 다른 객체에 위임해 보는 것은 어떨까.

### **뷰모델의 성질로 인해 강제되는 긴 테스팅은 개발자를 지치게 만든다.**

MVVM의 뷰모델은 단적으로 객체지향프로그래밍의 클래스이다. 근본적으로 결과는 속성을 통해 확인되고 메서드는 속성을 수시로 조작하는 권한을 갖는다. 이러한 조건 속에서 테스팅은 객체지향프로그래밍에 필연적으로 따른다. 하지만 그조차도 결함을 영원히 없앨 수는 없다. 이렇듯 끝이 나지 않는 과제 때문에 생산성은 저해될 수 있다.

결함의 근본적인 원인인 사이드이펙트를 없앨 수만 있다면 이상적일 것이다. 하지만 사이드이펙트 없는 프로그램은 존재할 수 없다. 중요한 건 사이드이펙트가 반드시 필요한 지점에서만 그것을 사용하는 일이다. 지금의 뷰모델에는 온갖 책임이 뒤섞여 있으므로 정확한 진단이 어렵다. 우선은 책임을 적절한 구분에 따라 분리한 다음 사이드이펙트를 특정한 위치로 격리시켜 관리할 수 있어야 한다.

분명하게도 사이드이펙트는 여전히 관리되어야 한다. 테스트 코드가 쓸모없지 않다는 말이다. 하지만 거꾸로 말해서 사이드이펙트가 발생하지 않는 대목에서 테스트는 필요하지 않다. 그렇게나 복잡한 뷰모델을 다음과 같이 만들 수 있다면 어떨까?

```js
const add = (a + b) => {
  return a + b
}
```

이만 마친다.
