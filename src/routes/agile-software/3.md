지난 세 차례의 프로젝트에서 나는 MVVM 아키텍처를 채택했다. MVVM은 매력적인 아키텍처다. 다음과 같은 분명한 장점들이 있다.
1. **구조가 단순하다.** MVVM 아키텍처는 소프트웨어를 최소한도의 계층으로 분리한다. 이용자가 당장 마주하는 뷰(View)와 그것에 대응하는 정보의 현상태로서 뷰모델(ViewModel)이 있으며, 인터페이스를 통해 가려지는 정보의 원상태(Model)가 있다. 개발자는 개념에 맞게 코드를 분산시켜 소프트웨어의 유지보수를 보다 간편하게 만들 수 있다. 이는 계층의 개념적 구분이 비교적 명료한 까닭에 그다지 어려운 일이 아니다.
2. **가장 분리되어 있다.** MVC. MVP. 그리고 MVVM. 보이는 것처럼 모델과 뷰의 관계보다는 뷰와 컨트롤러, 프레젠터, 혹은 뷰모델과 같은 개념들 사이의 관계가 중요하다. MVC 뷰와 모델 사이의 협착이 있다. MVP에서 뷰모델은 자신의 변화를 알려야 하며, 뷰는 변화를 수용할 메서드를 추가적으로 구현해야 한다. 역시나 의존 관계가 있다. 그러한 의존 관계를 MVVM은 흔히 이야기하는 데이터 바인딩을 통해 해결한다. 핵심은 뷰모델이 정보를 뷰의 관점에서 있는 그대로 이용할 수 있는 형태로 저장하고, 뷰는 정보를 구독하는 것이다.
3. **테스트에 최적화되어 있다.** MVVM 아키텍처의 진가가 여기에 있다. 변화하는 부분은 검증되어야 한다. 무언가를 테스트하기 위해선 우선 대상을 정보의 형태로 가공해야 한다. 따라서 MVVM은 뷰에 대한 모델인 뷰모델을 만든다. 뷰모델이 이용자가 마주하는 뷰에 대한 근사치를 표현하므로 개발자는 까다로운 뷰의 테스트에서 해방될 수 있다. 대부분은 뷰모델만 테스트하면 된다. 나아가 순전히 정보의 올바른 변화만을 관찰하기 위해 기반 시스템의 구현부를 외부 의존성으로 추출한다. 의존성들을 가짜 객체로 치환하면 소프트웨어를 실제로 구동하지 않고서도 여러가지 상황을 빠르게 시뮬레이션할 수 있다.

이처럼 분명한 장점들에도 불구하고 직접 구현하면서 느낀 단점들도 많다. 때때로 위의 특성들 자체가 한계를 보여주기도 한다. 글을 적는 목적은 보다 나은 설계를 위해 우선 문제점을 분명히 인식하고 해결의 실마리를 이끌어 내기 위해서이다.

## MVVM 아키텍처는 사실 단순하지 않다.
### 더 구체적으로 생각해 보기 1: 하나의 뷰에 여러 개의 뷰모델이 필요한 경우
일반적으로는 모바일 애플리케이션과 궁합이 좋다. 하나의 화면을 하나의 뷰로서 취급한다면, 하나의 화면마다 하나의 뷰모델을 둘 수 있다. 그러면 소프트웨어의 구조는 분명히 단순해진다.
```
+-- screens
|   +-- home
|       +-- viewmodel
|       +-- view
|   +-- signin
|       +-- viewmodel
|       +-- view
|   +-- profile
|       +-- viewmodel
|       +-- view
.
.
.
```
한 화면에 하나의 기능. 이 세련된 원칙을 지킬 수 있다면 위의 구조는 이상적이다. 모바일 애플리케이션에서는 이러한 일이 많다. 하지만 보다 넓은 지면을 활용하는 웹에서는 그렇지 않다. 상이한 성격을 가진 여러 요소들이 하나의 화면 위에 공존한다. 물론 모바일 애플리케이션도 아주 예외는 아니다. 홈 화면에는 종종 다종다양한 컴포넌트들이 끼어든다. 

멀리 가지 않더라도 하단 네비게이션을 가지고 있는 익숙한 화면들의 경우가 그렇다. 하단 네비게이션 뷰에게도 현재 상태를 나타낼 모델은 필요하다. 이를테면 현재 위치를 가리키는 색인 값이 그것이다. 색인 값이 없다면 현재 위치를 나타내는 아이콘을 하이라이트할 수 없을 것이다. 나아가서 채팅 등의 새로운 피드를 받는 애플리케이션이라면 읽지 않은 피드의 개수 또한 가지고 있어야 한다.

이때 역시 화면 단위로 뷰모델을 구성한다면 어떤 일이 생길까? 가능하지만 중복이 생긴다. MVVM 아키텍처에서 뷰모델은 비즈니스 로직을 포함하는 메서드를 함께 구현하게 마련이다. 그러한 관례를 따르자면 우선 하단 네비게이션의 아이콘을 클릭했을 때 작동할 코드를 화면마다 다시 작성하는 수고를 감내해야 한다. 예를 든 것처럼 피드를 받아야 한다면 어떨까? 매 화면의 뷰모델마다 다시 처음 상태를 불러오고, 계속 현재 상태를 갱신할 수 있도록 다시 웹소켓을 연결해 주어야 할 것이다. 이건 단순히 개발자를 고생시키는 의미를 넘어선다. 즉 네트워크 자원을 낭비하게 되는 것이다.

이렇게 되면 조금의 재치를 발휘해서 하단 네비게이션과 같이 공통된 컴포넌트를 독립적인 단위로 취급하고 싶어진다. 아마 다음과 같이 구조를 잡으면 될지도 모른다.

```
+-- components
|   +-- bottom_navigation
|   |   +-- viewmodel
|   |   +-- view
|   +-- about
|   |   +-- viewmodel
|   |   +-- view
|   +-- signin_form
|   .   +-- viewmodel
|   .   +-- view
|   .
```
일단 최상위의 범주를 components라고 변경해야 했다. 하단 네비게이션이 화면은 아니니까 말이다. 기왕의 화면들도 컴포넌트의 개념에 맞게 재정의했다. 디렉토리 구조는 여전히 깔끔하다. 하지만 이렇게 추출된 하단 네비게이션 컴포넌트는 어떻게 다시 삽입되어야 할까? 두 가지 방법이 떠오른다.

```dart
class HomeView extends StatelessWidget {
  final viewModel = Dependency.inject<HomeViewModel>();

  @override
  Widget build(BuildContext context) {
    return Scaffold(
      body: ..., // TODO: Use the HomeViewModel
      bottomNavigationBar: BottomNavigationView(),
    );
  }
}
```
UI를 그리기 위해 선언적 구문을 사용하는 Flutter 스타일로 작성하였지만 예컨대 안드로이드도 마찬가지다. 디렉토리 구조상으로 동일한 층위에 나란히 있는 두 컴포넌트가 실은 중첩된(Nested) 관계를 맺는다. `HomeView`는 `BottomNavigationView`에 의존하는 것처럼 보인다. 다른 방법도 있다.
```jsx
const HomePage = () => {
  return (
    <div>
      <header>
        <TitleComponent />
        <NavigationComponent />
      </header>
      <main>
        <AboutComponent />
      </main>
      <aside>
        <AdComponent />
      </aside>
      <footer>
        <FooterComponent />
      </footer>
    </div>
  )
}
```
조금 더 복잡한 화면의 예시를 들었다. Component를 Flat하게 나열할 수 있는 Wrapper가 필요한 것이다. 물론 이 Wrapper는 또한 가시성을 가지고 있는 View라고 말할 수도 있다. 일반적으로 컴포넌트들이 아무렇게나 배치되어 있지는 않기 때문이다. 하지만 ViewModel을 가지고 있지 않은 View라면 MVVM에서 큰 의미를 가지지는 않는다. 독립적인 요소라고 생각해도 좋을 것이다.
```
+-- app
|    +-- pages
|    |   +-- home
|    |   +-- signin
|    |   .
|    |   .
|    |   .
|    +-- components
|        +-- bottom_navigation
|        |   +-- viewmodel
|        |   +-- view
|        +-- about
|        .   +-- viewmodel
|        .   +-- view
|        .
```
디렉토리 구조는 아마도 이와 비슷한 모습이 된다. 나름 정리가 된 것처럼 보인다.

### 더 구체적으로 생각해 보기 2: 여러 개의 뷰에 하나의 뷰모델이 필요한 경우

지금까지는 한 화면에서 여러 개의 뷰모델이 필요한 일반적인 경우에 대해 고려했다. 또 다른 일반적인 경우가 있다. 여러 개의 화면을 넘나드는 와중에 하나의 뷰모델이 필요한 경우들이다. 구글에서 친절하게 마련한 안드로이드 튜토리얼에서는 컵케이크를 주문하는 플로우를 예시로 들고 있다.

한 화면에 하나의 기능. 하지만 기능의 구획은 매번 임의적일 수밖에 없다. 컵케이크를 주문하기. 이건 하나의 기능이다. 하지만 그러한 과정 속에 있는 가령 주문 종류와 수량 입력하기, 배송지 입력하기 등도 역시 각각 하나의 기능으로 여겨질 수 있다. 하지만 그처럼 작은 단위마다 뷰와 뷰모델을 구현하면 문제가 발생한다. 뷰모델은 다른 뷰모델을 의존할 수 없으므로 상태가 공유되지 않는 것이다.

이러한 문제를 해결하기 위해 구글의 안드로이드 튜토리얼에서는 SharedViewModel이라는 개념을 제안한다. 뷰에 대해 1:n의 관계를 가지는 뷰모델이다. 이 뷰모델은 뷰보다 긴 생명주기를 가질 수 있다. 직관적이고 간단한 해결책이다. 그런데 소프트웨어의 구조를 나타내는 디렉토리 구조상에서는 어디에 위치해야 하는 걸까?
```
+-- app
|    +-- pages
|    |   +-- home
|    |   +-- order1
|    |   +-- order2
|    |   .
|    |   .
|    |   .
|    +-- components
|        +-- bottom_navigation
|        |   +-- viewmodel
|        |   +-- view
|        +-- about
|        |   +-- viewmodel
|        |   +-- view
|        +-- order
|        .   +-- viewmodel
|        .   +-- views
|        .       +-- view1
|                +-- view2
|                .
|                .
|                .
```
바로 뷰들의 위치가 이동한다. 적힌 order는 더 이상 컴포넌트라고 부르기 애매할지도 모른다. 그도 그럴 것이 order는 차라리 컴포넌트들의 집합일 것이기 때문이다. 적어도 가져오는 pages의 입장에서 order는 여러가지 형태를 띤다. 하지만 이름 정도야 큰 문제는 아니다.

SharedViewModel은 문제를 아주 쉽게 해결하는 것처럼 보인다. 하나의 뷰모델이 여러 개의 View를 감싸는 방법을 통해서 말이다. 하지만 그만큼 뷰모델은 비대해진다. 그럴 수밖에 없는 것은 재차 말하지만 뷰모델은 테스트 가능하기 위해서 어떤 것도 의존하지 않아야 하기 때문이다.

심각하게 체감하기 위해서 아주 일상적으로 경험하는 이용자 정보에 대한 예를 들어 보자. 물론 프로필 화면으로 이동하면 이용자의 정보를 전체적으로 보여주겠지만, 종종 상단 네비게이션 바의 우측에도 조그마한 프로필 아바타가 위치한다.

일반적으로 정보의 원천은 같다. Restful API를 구현한 백엔드와 통신한다면 GET /users/:id를 통해 이용자 정보를 불러올 필요가 있다. 네비게이션 바와 프로필 화면이 서로 다른 컴포넌트인 까닭에 독립적인 뷰모델을 가지고 있다면 당연히 통신은 두 번 일어나야 한다. 곱절로 소모되는 네트워크 자원이 아까워서 이번에도 두 컴포넌트를 감싸는 거대한 뷰모델을 만들면 만사가 쉽게 해결될까? 결코 그렇지 않다. 마지막엔 뷰모델에 애플리케이션 전체를 아우르는 비즈니스로직을 몽땅 밀어넣고 나서야 끝날 것이다.

어떻게 할 것인가? 그동안 나는 아키텍처를 무너뜨리는 편보다는 네트워크 비용을 감수하는 쪽을 택했다.

## 알다시피 뷰모델은 과도한 책임을 떠안는다.
이른바 Massive View Controller는 여기에도 있다. 근본적으로 달라진 것이 딱히 없는 탓이다. 방금 살펴 본 SharedViewModel만 참고하더라도 알 수 있다. 뷰모델은 당초에 뷰에 대한 모델이었다. 핵심은 뷰가 뷰모델의 데이터를 있는 그대로 획득(get)할 수 있어야만 한다는 것이다. 하지만 SharedViewModel은 이미 뷰모델을 상태를 저장하는 용도까지 확장하고 있다. 예를 들어 안드로이드 Kotlin 코드를 보자.

```kt
class ProfileViewModel: ViewModel {
  private val user = MutableLiveData<UserModel>()

  val username: LiveData<String> get() = MediatorLiveData<String>().apply {
    addSource(user) { value = it.username }
  }

  val avatarImageUri: LiveData<String> get() = MediatorLiveData<String>().apply {
    addSource(user) { value = it.avatarImageUri }
  }
}
```

물론 이러한 형태가 아주 간편하다. 뷰가 있는 그대로 받아야 할 username과 avatarImageUri 획득자 속성은 다시금 user라는 상태를 구독한다. 주어진 예처럼 뷰모델은 그러한 현재 상태를 여러가지 형태의 획득자 속성을 위해 미리 준비할 수도 있지만, 다시 스스로 상태를 사용할 수도 있다. 그리고 그건 아주 흔한 일이다. 왜냐하면 일반적으로 뷰모델은 이용자에게 출력할 획득자 속성뿐만이 아니라 이용자의 입력을 수신하는 메서드도 함께 가지고 있기 때문이다.

```kt
class SignInFormViewModel(
  private val authClient: AuthClient,
): ViewModel {
  private val _email: MutableLiveData<String>("")

  val email: LiveData<String> get() = _email;

  private val _password_: MutableLiveData<String>("")

  val password: LiveData<String> get() = _password;

  fun onEmailChanged(value: String) {
    _email.value =  value;
  }

  fun onPasswordChanged(value: String) {
    _password.value = value;
  }

  fun onSubmitted() {
    val email = _email.value;
    val password = _password.value;

    val result = authClient.signIn(email, password)

    ...
  }
}
```
아마도 이용자가 이메일과 비밀번호를 입력한 뒤에 제출버튼을 누르면 `onSumibtted()` 메서드가 실행될 것이다. 이때 메서드는 앞서 저장된 상태를 사용한다. 인자를 통해 이메일과 비밀번호를 받지 않는 이유가 있다.
1. **메서드를 호출하는 뷰가 인자를 전달하기 어렵다.** HTML에는 form element가 있어서 해당하는 element 내부에 입력된 값들을 제출한 시점에 모두 읽어들일 수 있다. 하지만 모든 프레임워크에 존재하는 일반적인 기능은 아니다. 뷰는 무상태성을 가져야 하며 일종의 순수 함수여야 한다. 다시 말해서 나중에 전달할 상태를 저장하고 있어서는 안 된다.
2. **뷰로 내보낸 데이터는 모델 단에서 이용하는 형태와 차이가 있을 수 있다.** 이건 모델을 굳이 뷰모델의 획득자 속성으로 변환하는 이유이기도 하다. 이를테면 패스워드는 `*`와 같이 특정한 기호로 은닉되어 뷰로 내보내야 할 수 있다. 또한 `10000`처럼 정수로 존재하는 금액은 `"10,000"`와 같이 적절하게 변환하여 내보내야 한다. 인자를 전송해야 할 뷰에게는 데이터의 원형이 지워진다.
