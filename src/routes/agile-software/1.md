검색을 잘 못하는 건지는 모르겠지만 공개된 자료가 좀처럼 보이지 않았다. 다들 어떻게 해결하고 있는지 알 수 없다. 어쨌든 해결해야 할 문제가 있고 구체적인 목표가 있으니 나름대로 생각해 볼 수는 있다.

## 애플리케이션 단에서 발생하는 동시성 문제
이를테면 회원가입을 생각해 보자. 먼저 이메일과 비밀번호의 유효성을 검사해야 할 것이다. 다음으로는 주어진 이메일로 가입된 유저가 이미 존재하는지를 판별해야 한다. 모든 것이 통과하면 새로운 유저를 저장소에 저장해도 좋을 것이다. 마지막으로는 이용자의 아이디와 이메일을 반환해 주도록 한다. 코드를 작성해 보자.

```javascript
app.post("/", (req, res) => {
  const { email, password } = req.body

  if (isInvalidEmail(email)) {
    res.status(400).send({ message: "이메일이 유효하지 않습니다." })

    return
  }

  if (isInvalidPassword(password)) {
    res.status(400).send({ message: "비밀번호가 유효하지 않습니다." })

    return
  }

  const user = userRepository.findByEmail(email)

  if (user) {
    res.status(409).send({ message: "이미 존재하는 이메일입니다." })

    return
  }

  const { id, email } = userRepository.save(email, password)

  res.json({
    id,
    email,
  })
})
```

위의 코드는 주어진 요구사항을 있는 그대로 반영하고 있다. 구현부가 올바르다면 실제로 잘 작동한다. **거의 동시에 2개 이상의 요청을 받지 않는다면 말이다.**

문제는 바로 이것이다. 시험삼아 무수한 회원가입 요청을 날려 보자. 코드만 보자면 중복된 이메일을 가진 이용자가 없어야 할 것만 같지만, 실상은 그렇지 않다. 저장소에서 이용자의 존재 여부를 판별하기 위해 결과를 가져오는 코드와 새로운 이용자를 저장하는 코드 사이에는 시간차가 있다. 시간차 때문에 복수의 요청이 존재 여부를 판별하는 조건문을 통과할 수 있는 것이다. 바로 그 찰나의 시간 안에 요청을 밀어 넣을 수 있을 만큼 똑같은 이메일을 가진 유저는 생성된다.

이건 애플리케이션 단에서 일어나는 문제다. 바로 그곳에 조건문이 걸려 있기 때문이다. 그리고 비단 회원가입에 관한 문제에 국한되지 않는다. 영속적인 저장소와 같은 공유상태에 의존하는 어떤 비즈니스 로직이라도 위의 문제에 직면할 수 있다.

## 가설 1. 단 하나의 Queue를 만들기
가장 먼저 떠오르는 해결책은 단 하나의 Queue를 만드는 방법이다. 당연하게도 문제는 복수의 요청이 동시에 수행되기 때문에 발생한다. 따라서 모든 요청을 들어오는 대로 차곡차곡 쌓아서 하나씩 처리하면 되지 않을까?

```javascript
app.post('/', (req, res) => {
  // TODO: 매개변수의 유효성 검사를 수행한다.

  const params = {
    email,
    password,
  }

  const job = SignUpUseCase(params)

  const observer = usecaseQueue.enqueue(job)

  observer.onComplete = (result) => {
    const { id, email } = result

    res.status(201).json({
      id,
      email,
    })
  }
})
```

이용자가 거의 없는 서비스라면 아마도 문제가 없을 것이다. 하지만 조금이라도 이용자가 생긴다면 이 해결책은 서버의 성능을 과도하게 저하시킨다. 이를테면 어떤 사용자의 구매 요청은 다른 이용자의 파일 업로드 요청 때문에 지연될 수 있다. 실용성이 없는 방법이다.

## 가설 2. 요청마다 Queue를 만들기
그렇다면 요청마다 각기 다른 Queue를 만들면 될까? 별개의 요청이 서로에게 간섭하지 않도록 만들기 위해서 말이다.

하지만 오히려 서로를 의식해야 하는 요청들도 있다. 예를 들어 보자. 요청1이 있다. 이 요청은 유저의 포인트를 소모하여 상품을 구매한다. 요청2가 있다. 이 요청은 유저의 포인트를 현금으로 환전한다. 서로 다른 요청이지만 둘 모두가 포인트라는 상태를 공유한다. 처음 봤던 동시성 문제는 여기에서도 똑같이 일어날 수 있다.

반면에 같은 요청이지만 충돌할 여지가 없을 수도 있다. 이를테면 블로그의 포스트를 반환하는 요청이 있다고 하자. 하나의 요청은 다른 요청을 기다릴 필요가 없다. 이 경우에도 동일한 상태를 공유하고 있지만 말이다.

따라서 요청마다 Queue를 만드는 방법도 문제를 해결하지 못한다.


## 공유상태에 관하여

차라리 주목해야 할 대상은 **공유상태**라고 가정했다. 공유상태는 말 그대로다. 저마다의 요청이 공유하는 상태성을 뜻한다. 데이터베이스에 담긴 정보가 대표적인 예시다. 그러한 상태성은 비즈니스 로직 내부에서 단 두 가지 형태로 사용할 수밖에 없다.

1. 공유상태를 읽는다.
2. 공유상태를 쓴다.

공유상태의 읽기에 대한 용례를 살펴 보자.

```javascript
app.get("/", (req, res) => {
  const userId = ...

  const user = userRepository.find(userId);

  if (!user) {
    res.status(404).send({ message: "이용자를 찾지 못했습니다." })

    return
  }

  const results = articleRepository.findAllByUserId(user.id);

  res.json(results);
});
```

위의 코드는 결과값을 읽은 다음 반환하기에 앞서 이용자의 존재 여부를 판별한다. 위에서는 공유상태를 읽고만 있다. 동시성 문제가 발생할 소지가 있는지 검토해 보자.

1. **동시에 여러 번 동일한 용례를 요청한다**

이 경우에 문제가 발생할 일은 없을 것이다.

2. **나중에 요청한 어떤 용례가 위의 코드가 진행되는 도중에 공유상태를 읽는다.**

읽기는 어느쪽이든 전혀 영향을 끼치지 않는다.

3. **나중에 요청한 어떤 용례가 판별식이 통과한 이후에 공유 상태를 조작한다**

물론 이용자는 오래된 결과를 볼 것이다. 하지만 이건 HTTP 프로토콜의 성질로 간주할 수 있다.

4. **나중에 요청한 어떤 용례가 판별식이 통과한 이후에 판별식의 결과를 바꿀 만한 조작을 수행한다** 

이용자는 현재로서는 보아서는 안 될 결과를 관측할 것이다. 그렇지만 마찬가지로 HTTP 프로토콜의 성질로 간주할 수 있다. 읽기는 봐 줄 만한 위험이라는 생각이 든다.

다음은 공유 상태의 쓰기에 대한 용례를 살펴 보자.

```javascript
app.get("/", (req, res) => {
  const username, password = ...

  const user = userRepository.save(username, password)

  res.json(user);
});
```

위의 코드는 단순히 새로운 유저를 저장한다. 오로지 쓰기만을 위해 중복검사를 하지 않았다. 판별하기 위해서는 공유상태를 읽어야 하는 탓이다. 역시 동시성 문제가 발생할 소지가 있는지 검토해 보자.

1. **동시에 여러 번 동일한 용례를 요청한다**

이 경우에 문제가 발생할 일은 없을 것이다. 지금은 애플리케이션 단의 동시성 문제를 다루고 있다.

2. **나중에 요청한 어떤 용례가 위의 코드가 진행되는 도중에 공유상태를 쓴다.**

마음껏 쓰라고 하자. 다른 용례의 쓰기를 막을 만한 이유는 아무것도 없다.

2. **나중에 요청한 어떤 용례가 위의 코드가 진행되는 도중에 공유상태를 읽는다. 그 용례는 읽기만 한다.**

공유상태를 쓰기 이전에 읽든지, 이후에 읽든지 큰 위험은 없다.

3. **나중에 요청한 어떤 용례가 위의 코드가 진행되는 도중에 공유상태를 판별한다. 그 용례는 쓰기도 한다.**

아마도 문제가 생길 수 있을 것이다. 이를테면 어떤 용레가 중복검사를 수행한 다음 이용자를 생성하는 또다른 용례인 A라고 하자. 용례 A가 중복검사를 하고 나서 이용자를 생성하기 전에 위의 용례가 동일한 이름을 가진 이용자를 먼저 저장한다면, 여기서 충돌이 발생한다.

## 쓰기 위해 읽어야만 할 때

문제는 바로 **쓰기** 위해 **읽어야만** 할 때다. 읽는 것은 또한 판별하기 위해서다. 잔가지를 쳐내고 다시 처음으로 돌아가자.

```javascript
app.post("/", (req, res) => {
  // TODO: 매개변수의 유효성을 검사한다.

  const user = userRepository.findByEmail(email)

  if (user) {
    res.status(409).send({ message: "이미 존재하는 이메일입니다." })

    return
  }

  const result = userRepository.save(email, password)

  // TODO: 응답한다.
})
```

위의 코드에서 새로운 이용자를 영속적인 저장소에 보관하기 위해서는 먼저 이용자를 불러와서 판별해야 한다. 진행 순서상 **쓰기**는 조건식을 작성하기 위한 **읽기**에 뒤따르는 것처럼 보이지만, 개발자의 계획에 있어서 앞서는 쪽은 오히려 **쓰기**다.

하나의 의심을 추가해 보자. 위의 코드에서 결국 저장할 이용자 객체는 중복을 검사하기 위해 시험 삼아 가져온 이용자 객체를 선점하는 것처럼 보인다. 동시성 문제가 쓰기 위해 읽어야만 할 때라면 발생한다면, 쓸 객체와 읽을 객체는 위의 예처럼 항상 밀접한 관련을 가져야만 하는 걸까? 반례를 생각해 보자.

```javascript
app.post("/", (req, res) => {
  // TODO: 매개변수의 유효성을 검사한다.

  const user = userRepository.find(userId)

  if (!user) {
    res.status(404).send({ message: "이용자를 찾지 못했습니다." })

    return
  }

  const { point } = user

  const amount = 400

  if (point < amount) {
    res.status(409).send({ message: "포인트가 부족합니다." })

    return
  }

  const { id, createdAt } = orderRepository.save(user.id, itemId)

  userRepository.updatePoint(user.id, point - amount)

  res.json({
    id,
    createdAt,
  })
})
```

위의 코드를 포인트로 새콤달콤을 주문하는 용례라고 생각해도 좋다. 우리의 과제를 해결하지 못하면 이때에도 커다란 문제가 생긴다. 나라면 포인트의 양을 판별하는 조건문과 일정한 양만큼 포인트를 감소시키는 구문 사이에 최대한 많은 요청을 밀어넣어 단돈 400포인트로 100개의 새콤달콤을 합법적으로 요구할 것이다.

실제로 읽는 값과 쓰는 값은 밀접한 관련이 있다. **즉 조건식이 허용한 쓰기는 그 조건식을 부정하는 쪽으로 작용해야 한다.** 이때 쓰기는 다름 아닌 자신인 어떤 값을 쓰는 일이다. 따라서 조건식에는 자기가 속해있을 수밖에 없다.

그림이 한결 명확해진 성싶다. **즉, 공유상태의 쓰기와 그로 인해 부정될 조건문 사이에 어떤 쓰기도 허용해서는 안 된다.** 몇 가지 예를 더 들어 보자.

```javascript
app.post("/", (req, res) => {
  const stocks = stocksRepository.findAll();

  if (stocks.isEmpty) {
    res.status(404).send({ message: "재고가 남아 있지 않습니다." })

    return
  }
  
  const order = orderRepository.save(stocks.first.id)

  stocksRepository.delete(stocks.first.id)

  res.json(order)
})
```

위의 코드는 주문을 위해 먼저 재고의 개수를 판별한다. 주문을 생성했다면, 첫 번째 재고를 하나 없앤다. 어떤 다른 요청이 위의 코드의 4번째 라인과 12번째 라인 사이에 재고 하나를 감소시킨다면 문제가 발생할 것이다. 반면 재고 하나를 추가하는 일은 문제가 없다.

```javascript
app.post("/", (req, res) => {
  const character = characterRepository.find(id);

  if (character.healthPoint == 0) {
    res.status(404).send({ message: "이미 처치했습니다." })

    return
  }

  const attackPoint = expensiveCalculation()

  const healthPoint = Math.max(character.healthPoint - attackPoint, 0)

  const characterRepository.updateHealthPoint(id, healthPoint)

  ...
})
```

위의 코드는 상대방의 캐릭터를 공격하는 용례인 것처럼 보인다. 상대방의 체력이 바닥났을 때는 공격하지 않는다. 어떤 다른 요청이 위의 코드의 3번째 라인과 10번째 라인 사이에 체력을 0으로 감소시키면 어떻게 될까? 위의 코드가 체력을 0으로 감소시키지 못한다면 먼저 실행된 다른 요청의 결과를 덮어씌울 것이다. 위의 용례에서는 다른 요청이 체력을 증가시키는 경우에도 마찬가지의 문제가 발생한다. 스코프 내부에서 character와 같은 공유상태는 실제 상태에 대한 불변성까지 보장하지는 않는 것이다.
