결국 나는 다음과 같은 코드를 쓰고 싶어졌다.

```javascript
app.post("/", (req, res) => {
  stockRepositoryMutex.lock()

  const stocks = stockRepository.findAll();

  if (stocks.isEmpty) {
    res.status(404).send({ message: "재고가 남아 있지 않습니다." })

    return
  }
  
  const order = orderRepository.save(stocks.first.id)

  stockRepository.delete(stocks.first.id)

  stockRepositoryMutex.unlock()

  res.json(order)
})
```

`StockRepositoryMutex`는 `StockRepository`가 기리키는 저장소 전체를 제어한다. 이 경우에 문제는 저장소의 총량이기 때문이다. 이 규칙으로 인해 다른 용례에서도 공유상태를 조작하기 위해서는 적절한 `Mutex`를 참조해야 할 것이다.

위의 코드에서는 `OrderRepository`도 참조한다. 혹시 모를 다른 용례를 의식하여 이 경우에도 `Mutex`를 사용해야 할 것이다.

```javascript
app.post("/", (req, res) => {
  ...

  const orderRepositoryMutex = orderMutex.getRepository()
  
  orderRepositoryMutex.lock()
  const order = orderRepository.save(stocks.first.id)
  orderRepositoryMutex.unlock()

  ...
})
```

`OrderRepoisotry`에 대한 조작도 `Mutex`로 감쌌다. 여기에 더해 잠글 대상도 정할 수 있게 하였다. 항상 저장소 전체를 제어해야만 하는 것은 아니기 때문이다. 다음과 같은 용례도 있다.

```javascript
app.post("/", (req, res) => {
  // TODO: 매개변수의 유효성을 검사한다.

  const userEntityMutex = userMutex.getEntity(userId)

  userEntityMutex.lock()
  const user = userRepository.find(userId)

  if (!user) {
    res.status(404).send({ message: "이용자를 찾지 못했습니다." })

    return
  }

  if (user.point < 400) {
    res.status(409).send({ message: "포인트가 부족합니다." })

    return
  }

  orderRepositoryMutex.lock()
  const { id, createdAt } = orderRepository.save(user.id, itemId)
  orderRepositoryMutex.unlock()

  userRepository.updatePoint(user.id, user.point - 400)

  userEntityMutex.unlock()

  res.json({
    id,
    createdAt,
  })
})
```

이 경우에는 저장소 전체가 아니라 저장소가 가지고 있는 하나의 객체만 제어하면 된다. 다만 앞선 저장소 전체의 제어권과의 관계를 주목해야 한다. 저장소 전체가 잠궈져 있는 동시에, 하나의 객체에 대한 제어를 얻을 수 있어서는 안 된다. 반대도 마찬가지다.

위의 코드와 관련해서 다른 용례를 하나 더 생각해 보자. 이를테면 이용자의 체력을 회복시키지만 최대체력 이상일 때는 예외를 발생시키는 용례를 말이다. 마찬가지로 특정한 이용자 객체를 잠시 잠글 수 있다. 바로 여기가 오버헤드가 생길 수 있는 부분이다. 위의 용례도 특정한 이용자 객체를 잠근다. 두 용례는 서로 관련 없는 속성만을 참조하지만 다른 용례의 동시 실행을 지연시킬 것이다. 따라서 다음과 같이 코드를 상상해 볼 수 있다.

```javascript
app.post("/", (req, res) => {
  // TODO: 매개변수의 유효성을 검사한다.

  const pointPropertyMutex = userMutex.getEntity(userId).getProperty("point")

  pointPropertyMutex.lock()
  const user = userRepository.find(userId)

  if (!user) {
    res.status(404).send({ message: "이용자를 찾지 못했습니다." })

    return
  }

  if (user.point < 400) {
    res.status(409).send({ message: "포인트가 부족합니다." })

    return
  }

  orderRepositoryMutex.lock()
  const { id, createdAt } = orderRepository.save(user.id, itemId)
  orderRepositoryMutex.unlock()

  userRepository.updatePoint(user.id, user.point - 400)

  pointPropertyMutex.unlock()

  ...
})
```

포함 관계를 볼 수 있다. 저장소에 대한 `Mutex`는 객체에 대한 `Mutex`를 포함한다. 객체에 대한 `Mutex`는 속성에 대한 `Mutex`를 포함한다. 이는 나중에 실제로 구현할 때 염두에 두어야 할 특징이다.

`Mutex`는 어떤 자원을 사용할 때 자동적으로 걸리면 좋을 것이다. 모든 쓰기는 국소적으로 `Mutex`가 사용되어야 한다. 하지만 모든 읽기에 `Mutex`를 사용할 필요는 없다. 이전 글에서 이야기 했던 **쓰기를 위한 읽기**에만 해당되는 원칙이다. 따라서 최종적인 코드가 다음과 같다면 멋질 것이다. Rust로 작성하겠다.

```rust
fn do_something() -> impl Responder {
  ...

  let locked_point = user_repository
    .where(format!("id = '{}'", user_id))
    .select("point")
    .with_mutex()
    .lock()
    .unwrap()
    
  let point = match locked_point {
      Some(x) => x,
      None => return status(404).message("이용자를 찾지 못했습니다."),
    }
  
  if point < 400 {
    return status(409).message("포인트가 부족합니다.");
  }

  let order = order_repository.save(user.id, item_id)

  user_repository.update_point(user_id, point - 400);

  drop(locked_point)

  ...
}
```

위의 코드는 먼저 특정한 이용자의 `point` 속성을 `Mutex`에 담아 불러온다. 이때의 `point`는 차후에 쓰기를 위해 읽어야만 하는 속성인 탓이다. Rust에서 `Mutex`의 락은 스코프가 끝날 때 자동으로 해제된다. 하지만 성능상의 이점을 취하기 위해 쓰기가 끝나면 수동으로 해제한다. `order_repository`의 `save` 함수와 `user_repository`의 `update_point` 함수 주변에는 추가적인 코드가 필요하지 않다. 쓰기는 암묵적으로 적절한 범위의 락을 수행한다.

이 정도면 그럴 듯해 보인다. 실제 구현이 남았다.
