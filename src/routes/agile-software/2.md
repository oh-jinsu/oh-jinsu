검색을 잘 못하는 건지는 모르겠지만 공개된 자료가 좀처럼 보이지 않았다. 다들 어떻게 해결하고 있는지 알 수 없다. 어쨌든 해결해야 할 문제가 있고 구체적인 목표가 있으니 나름대로 생각해 볼 수는 있다.

모든 요청을 쓸어담는 Queue를 만드는 방법은 실용성이 없었다. 요청마다 하나의 Queue를 만드는 방법 또한 애당초의 문제를 해결하지 못했다.

## 공유상태에 관하여

차라리 주목해야 할 대상은 **공유상태**라고 가정했다. 공유상태는 말 그대로다. 저마다의 요청이 공유하는 상태성을 뜻한다. 데이터베이스에 담긴 정보가 대표적인 예시다. 그러한 상태성은 비즈니스 로직 내부에서 단 두 가지 형태로 사용할 수밖에 없다.

1. 공유상태를 읽는다.
2. 공유상태를 쓴다.

공유상태의 읽기에 대한 용례를 살펴 보자.

```javascript
app.get("/", (req, res) => {
  const userId = ...

  const user = userRepository.find(userId);

  if (!user) {
    res.status(404).send({ message: "이용자를 찾지 못했습니다." })

    return
  }

  const results = articleRepository.findAllByUserId(user.id);

  res.json(results);
});
```

위의 코드는 결과값을 읽은 다음 반환하기에 앞서 이용자의 존재 여부를 판별한다. 위에서는 공유상태를 읽고만 있다. 동시성 문제가 발생할 소지가 있는지 검토해 보자.

1. **동시에 여러 번 동일한 용례를 요청한다**

이 경우에 문제가 발생할 일은 없을 것이다.

2. **나중에 요청한 어떤 용레가 위의 코드가 진행되는 도중에 공유상태를 읽는다.**

읽기는 어느쪽이든 전혀 영향을 끼치지 않는다.

3. **나중에 요청한 어떤 용례가 판별식이 통과한 이후에 공유 상태를 조작한다**

물론 이용자는 오래된 결과를 볼 것이다. 하지만 이건 HTTP 프로토콜의 성질로 간주할 수 있다.

4. **나중에 요청한 어떤 용례가 판별식이 통과한 이후에 판별식의 결과를 바꿀 만한 조작을 수행한다** 

이용자는 현재로서는 보아서는 안 될 결과를 관측할 것이다. 그렇지만 마찬가지로 HTTP 프로토콜의 성질로 간주할 수 있다. 읽기는 봐 줄 만한 위험이라는 생각이 든다.

다음은 공유 상태의 쓰기에 대한 용례를 살펴 보자.

```javascript
app.get("/", (req, res) => {
  const username, password = ...

  const user = userRepository.save(username, password)

  res.json(user);
});
```

위의 코드는 단순히 새로운 유저를 저장한다. 순전히 쓰기만을 위해 중복검사를 하지 않았다. 판별하기 위해서는 공유상태를 읽어야 하는 탓이다.
